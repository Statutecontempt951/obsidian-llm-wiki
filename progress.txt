### MCP vault.list root bug (dispatched to Codex, cold-verified by Claude)

- Repro: `vault.list({path:""})` -> "path required";
  `vault.list({path:"."})` -> "path traversal blocked"
- Root cause: `resolvePath` rejected empty + `.` before allowing either as
  "list vault root". Same bug lived in both `src/index.ts` (stdio MCP) and
  `src/connector/fs-transport.ts` (connector transport) -- Codex found and
  fixed both.
- Fix: split `resolvePath` into `normalizeVaultPath(opts: {allowRoot?})` +
  `resolve(opts: {allowRoot?})`; `vault.list` opts into `allowRoot`, other
  tools stay strict. Also replaced `full.startsWith(vault)` guard with
  `path.relative(vault, full).startsWith('..')` -- correct idiom on
  case-insensitive Windows FS, free hardening.
- Tests: 7 new node --test cases in `src/connector/fs-transport.test.ts`
  (root list x3, traversal reject x3, nested unchanged x1). Total mcp-server
  tests: 77/77 pass (cold-verified via `npm test` independently of Codex).
- Smoke (cold-verified by Claude): JSON-RPC roundtrip of `vault.list` with
  `{"":...,".":...,"/":...,"./":...}` all return real vault root;
  `{..:...,"C:/Windows":...,"../":...,"../etc":...}` all still blocked.

Design decisions locked (see planning note
`C:/Users/Administrator/.claude/plans/1-stale-clever-glacier.md`
for full rationale; summary here):

- **Schema**: 6-field frontmatter, all required on write:
  generated-by / generated-at / agent / parent-query / source-nodes
  / status. Each field prevents a distinct class of future
  failure (who / when / what quality / why generated / from where
  / still valid).
- **Lifecycle**: exactly 3 legal transitions. draft->reviewed is
  manual (human only). draft->stale is automatic (gardener by age
  + non-AI-Output backlink test). reviewed->superseded is
  semi-automatic (gardener reports candidates, human confirms).
  Chose B->A (default trust + gardener sweep stale) over A->B
  (default suspicion + human promotes reviewed) because the
  maintenance cost axis of B->A is O(误判 rate) whereas A->B is
  O(total volume).
- **Directory**: {vault}/00-Inbox/AI-Output/{persona}/
  YYYY-MM-DD-{slug}.md with per-persona subdirs. MVP simple;
  schema carries enough info that graduating to flat + yaml-lookup
  is a no-code change later if data justifies it.
- **Stale thresholds**: architect=45d, gardener=30d, historian=180d,
  librarian=60d, catch-all=60d. Hardcoded for MVP. Rationale:
  architect ideas iterate, gardener reports are periodic,
  historian facts age slowly, librarian is synthesis-medium.
- **Backlink rule**: a file has a "real" backlink iff the source
  file's own frontmatter does NOT carry `generated-by`.
  AI-Output->AI-Output references are self-anchoring hallucination
  chains and do not count as anchor votes.

### Engineering notes for next editor

- `node` in Git Bash is aliased to `winpty node.exe`; winpty
  requires a TTY which the Bash tool doesn't provide. Use
  `node.exe` directly when invoking tests / generator scripts.
  `npm run build` and `npm run generate-tools-doc` both work
  because npm internally invokes node without the alias.
- The `import.meta.url === process.argv[1]` main() guard in
  index.ts is a dual-role enabler (importable + runnable). If
  future refactors move main() into a different module, keep the
  guard — any future test importing VaultFs (or any other class
  from index.ts) will hang indefinitely without it.
- Stale-sweep's in-place status flip uses a narrow regex scoped
  to the first `---...---` block. This preserves body content and
  other fm fields but assumes the status field is a single `status:
  <word>` line (not multiline YAML). If future schemas add
  multiline fields around status, the regex needs widening.

### Two real issues the smoke test surfaced

First smoke run against the default cwd (mcp-server/) quietly
returned an empty listing. Second run, with cwd=vaultRoot, crashed
on server startup. Both are documented here because they are
real, not test artefacts:

**Issue A -- loadConfig precedence shadow (design choice, not a bug):**
loadConfig() probes `./vault-mind.yaml` then `../vault-mind.yaml`
BEFORE honouring `VAULT_MIND_VAULT_PATH`. When the server spawns
from mcp-server/, the dev-workspace yaml at
`/d/projects/obsidian-llm-wiki/vault-mind.yaml` (points at
E:/knowledge) captures precedence and the env var is ignored. The
test works around this by running the child with cwd=vaultRoot
and dropping a minimal `vault-mind.yaml` in the seed dir.

**Issue B -- bundled pglite vector extension path (pre-existing bug):**
`@electric-sql/pglite/vector` resolves `new URL('../vector.tar.gz',
import.meta.url)` at runtime. When esbuild inlines the vector
module into bundle.js, `import.meta.url` becomes the bundle's
path, so `../vector.tar.gz` resolves to
`D:/projects/obsidian-llm-wiki/vector.tar.gz` (non-existent)
instead of pglite's ships-in-the-package `release/vector.tar.gz`.
With the dev yaml's filesystem-only adapter list this never
tripped, but any deploy that enables `vaultbrain` via the default
adapter list (`["filesystem", "memu", "gitnexus", "obsidian",
"qmd", "vaultbrain"]` at src/index.ts line 1108) will crash on
first VaultBrain init.

The smoke test side-steps B by writing `adapters: filesystem`
into the seed yaml. The underlying bundle bug is NOT fixed in
this session -- documented as P1 below.

### Why this matters

Before session 5, every test was either a unit test (pure TS) or
an adapter test (injects mocks). None of them exercised the stdio
framing, the SDK Server glue, the tool-name bridge, or the env /
yaml config loader. Issue B in particular is a shipping-artifact
bug that has been latent since the vaultbrain adapter was added
and would not have been caught by unit tests at any resolution.

### Known gaps (explicit, carried to next session)

P1 (real design/impl bugs):
- **Bundled pglite vector extension path**: fix options are
  (1) externalize `@electric-sql/pglite/vector` in the esbuild
